Messy spreadsheet in. Clean, structured data out.

Before (raw spreadsheet)

  "source": "LinkedIn",
  "stage": "new",
  "created_at": "2026-04-01"

Canonical schema. Clean date. Default stage applied. Ready for Sheets.

Why OpenAI? "Full Name" → name. "Business Name" → company. "April 1, 2026" → "2026-04-01". A language model handles this naturally. Regex would break on the first edge case.

Cross-Reference Reconciliation

This is where your data starts talking to itself.

New invoice arrives: { client: "Acme Corporation", amount: $4,500 }
    ▼ search Leads tab → Found: "Acme Corp" (stage: "contacted") → update stage → "converted"
    ▼ search Projects tab → Found: "Website Redesign" (client: "Acme Corp") → total_billed += $4,500

Fuzzy matching: "Acme Corporation" and "Acme Corp" are obviously the same company to you. And to OpenAI. But == says they're different. This is why we use a language model.

Every prompt to Claude Code follows one pattern:

Where you are and what's happening.

"I'm on the /invoices page"

Goal

The outcome you want.

"Add a Set Reminder button on overdue invoices"

Constraints

What NOT to change.

"Don't change the table layout"

"I'm on the /invoices page. Overdue invoices need a 'Set Reminder' button that creates a Google Calendar event for the due date. Use the existing calendar service and keep the table layout as-is."

When Claude gets it wrong — this is the best teaching moment. Say: "Not quite — use due_date, not created_at." The audience sees that iteration is normal, not failure.

Under the Hood

Clean Architecture

This is how the app stays organized — so when something breaks, you know exactly where to look.

Every request flows through four layers. Each layer has one job.

Route

"What URL was hit?" — Maps URLs to controllers. Thin wiring, nothing else.

Controller

"What did the user send?" — Reads the request, calls the right service, returns a response.

Service

"What should happen?" — Business logic, API calls, data processing. The brains.

Repository

"Where is the data?" — Reads and writes to files or Google Sheets. Data access only.

Route  →  Controller  →  Service  →  Repository

Why this matters: When something breaks, you know exactly which layer to look at. The route didn't match? The controller read the wrong field? The service called the wrong API? The repository wrote to the wrong file? One layer, one problem.
