What is a Code Editor?

A code editor is the app where you view and edit your project files. Think of it like Microsoft Word, but for code.

Visual Studio Code (VS Code)

A free code editor made by Microsoft. The most popular editor in the world. Over 70% of developers use it.

Download: code.visualstudio.com

Cursor

An AI-focused code editor built on top of VS Code. Looks and works the same way, with extra AI features built in.

Download: cursor.com

Why these two? They're free, beginner-friendly, and widely supported. There are other editors out there (Sublime Text, WebStorm, Vim) and Claude Code works in all of them. We chose these because they're the easiest starting point.

1

In your editor, go to File in the top menu bar, then click Open Folder. Navigate to your project folder and select it.

If you're starting from scratch:

First, create a new folder. Open Finder (Mac) or File Explorer (Windows). Go to your Desktop or Documents. Right-click, choose New Folder, and name it something like my-dashboard. Then go back to your editor and open that folder with File → Open Folder.

Why does this matter? When Claude Code starts, it looks at all the files in your open folder. That's how it knows what your project is. If you open the wrong folder, or no folder at all, Claude won't know what you're working on.

Install Claude Code

Now that Node.js is installed, one command gets you Claude Code.

Still seeing "claude: command not found"? This is the #1 issue. It means your system can't find where Claude Code was installed.

Mac fix: Run: echo 'export PATH="$HOME/.npm-global/bin:$PATH"' >> ~/.zshrc && source ~/.zshrc

Windows fix: Close PowerShell completely and reopen it. If still broken, restart your computer.

Stuck? Open claude.ai in your browser and start a new chat. Paste this: "I'm trying to install Claude Code on my [Mac/Windows]. I ran the install command but I'm getting [your error]. Help me fix it step by step." Claude can debug your specific setup.

34
Session 1: Homework

CLAUDE.md: The Master File

This is the ONLY file Claude Code reads automatically. Everything else, you have to point it to. Get this one right.

Do:

  • Start with what the project IS in 2-3 sentences
  • List your tech stack explicitly
  • Define your data shapes (Lead, Invoice, Project)
  • Describe your folder structure
  • List commands to run the app
  • State rules: "never commit .env"
  • Reference your other docs: "See Design.md for layouts"
  • Keep it under 2 pages

Don't:

  • Write a novel. Claude skims long files.
  • Be vague: "make it nice" means nothing
  • Skip the data shapes. Claude needs to know what a Lead looks like.
  • Forget the rules. Claude will happily commit your API keys if you don't say "never commit .env"
  • Duplicate info from other files. Point to them instead.
  • Use jargon without explaining it
35
Session 1: Homework

Design.md: The Visual Blueprint

This file tells Claude what your app looks like. Without it, Claude picks random colors, random layouts, and random components.

Do:

  • List every page and its purpose
  • Describe the layout: "metric cards on top, table below, chart at bottom"
  • Define your color palette with hex codes
  • Specify badge colors for each status
  • Describe what buttons exist and what they do
  • Include a sample Claude Code prompt for each page

Don't:

  • Say "make it look modern" with no specifics
  • Forget to define status badge colors. Claude will pick its own.
  • Skip the navigation structure
  • Leave out the empty states ("No leads yet. Sync your Drive.")
  • Assume Claude knows what "clean" or "professional" means to you

Pro tip: If you can describe it to a friend over the phone, you can describe it to Claude. "I want 4 boxes at the top showing my key numbers, then a table below with all my leads" is a perfect start.

Session 1: Homework

Web_Architecture.md: The Plumbing

This file maps how data moves through your system. From Google Drive to your dashboard to Gmail.

Do:

  • Draw a flow diagram (even in text/ASCII)
  • List every API and what it does
  • Explain the request flow: "user clicks button, Flask calls service, service calls API"
  • Document how authentication works
  • List every URL/route in your app
  • Explain why you made key decisions ("why Sheets instead of a database?")

Don't:

  • Skip the diagram. A picture is worth 1000 words, even an ASCII one.
  • Assume Claude knows which layer calls which
  • Forget to document the authentication flow
  • Leave out error scenarios ("what if the token expires?")
  • Mix up "what the app does" with "how the app is built"
37
Session 1: Homework

API_Design.md: The Recipe Book

This file documents every API call AND how your data objects relate to each other. It's the most detailed file.

Do:

  • Show the exact API call with code examples
  • Show a sample response (what comes back)
  • List common errors with plain-English fixes
  • Draw an object relationship diagram showing how Lead, Invoice, and Project connect
  • Define the linking field (company/client name) and explain why it's fuzzy-matched
  • Document the lifecycle: lead enters > project created > invoice uploaded > lead auto-converts
  • Explain design decisions ("why names instead of IDs?")

Don't:

  • Just link to external docs. Claude needs the details HERE.
  • Skip error handling. Claude will write happy-path-only code.
  • Forget object relationships. Without them, Claude builds isolated tables that don't talk to each other.
  • Leave out auto-update rules (who updates what when a new record arrives)
  • Skip the "why" behind design decisions. Claude makes better choices when it understands intent.

Key insight: The object relationships are MORE important than the API calls. Getting Lead > Invoice > Project linking right determines whether your dashboard feels like magic or a disconnected spreadsheet viewer.

Your homework is to draft all 4 files before Session 2. They don't have to be perfect. Start with bullet points. You can refine them with Claude: paste your draft and ask "review this and tell me what's missing."

OpenAI Normalization

Messy spreadsheet in. Clean, structured data out.

Before (raw spreadsheet)

Headers:
  "Full Name", "Email Address",
  "Business Name", "How Found",
  "Date Added"

Rows:
  "Jane Smith", "jane@acme.com",
  "Acme Corporation", "LinkedIn",
  "April 1, 2026"

Wrong column names. Inconsistent date format. Missing fields.

After (normalized)

{
  "name": "Jane Smith",
  "email": "jane@acme.com",
  "company": "Acme Corporation",
  "source": "LinkedIn",
  "stage": "new",
  "created_at": "2026-04-01"
}

Canonical schema. Clean date. Default stage applied. Ready for Sheets.

Why OpenAI? "Full Name" → name. "Business Name" → company. "April 1, 2026" → "2026-04-01". A language model handles this naturally. Regex would break on the first edge case.

Cross-Reference Reconciliation

This is where your data starts talking to itself.

New invoice arrives: { client: "Acme Corporation", amount: $4,500 }
    ▼ search Leads tab → Found: "Acme Corp" (stage: "contacted") → update stage → "converted"
    ▼ search Projects tab → Found: "Website Redesign" (client: "Acme Corp") → total_billed += $4,500

Fuzzy matching: "Acme Corporation" and "Acme Corp" are obviously the same company to you. And to OpenAI. But == says they're different. This is why we use a language model.

TriggerLookupAuto-Update
New invoiceMatch lead by clientLead stage → "converted"
New invoiceMatch project by clientProject total_billed += amount
New leadMatch project by companyLink lead to project
New projectMatch leads + invoicesAssociate existing records

How We Build

The Golden Formula

Every prompt to Claude Code follows one pattern:

Context

Where you are and what's happening.

"I'm on the /invoices page"

Goal

The outcome you want.

"Add a Set Reminder button on overdue invoices"

Constraints

What NOT to change.

"Don't change the table layout"

Full example:

"I'm on the /invoices page. Overdue invoices need a 'Set Reminder' button that creates a Google Calendar event for the due date. Use the existing calendar service and keep the table layout as-is."

When Claude gets it wrong, this is the best teaching moment. Say: "Not quite, use due_date, not created_at." The audience sees that iteration is normal, not failure.

Under the Hood

Clean Architecture

This is how the app stays organized, so when something breaks, you know exactly where to look.

Every request flows through four layers. Each layer has one job.

Route

"What URL was hit?" Maps URLs to controllers. Thin wiring, nothing else.

Controller

"What did the user send?" Reads the request, calls the right service, returns a response.

Service

"What should happen?" Business logic, API calls, data processing. The brains.

Repository

"Where is the data?" Reads and writes to files or Google Sheets. Data access only.

Route  →  Controller  →  Service  →  Repository

Why this matters: When something breaks, you know exactly which layer to look at. The route didn't match? The controller read the wrong field? The service called the wrong API? The repository wrote to the wrong file? One layer, one problem.
